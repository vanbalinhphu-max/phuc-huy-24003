URL: 
    địa chỉ website dùng để truy cập trên trình duyệt (https://interdata.vn/blog/seo-la-gi/)
    bao gồm scheme (ví dụ https), domain (interdata.vn), và path

Path (Đường dẫn):
    Là phần của URL đứng sau domain, dùng để xác định tài nguyên cụ thể đàng được yêu cầu
    (/blog/seo-la-gi/)
    Được tạo thành từ 1 hoặc segments

Route (Tuyến đường):
    Trong nextjs, một route là 1 đường dẫn (path) trong ứng dụng mà người dùng có thể truy cập
    Nó được định nghĩa bởi cấu trúc thư mục bên trong thư mục app và các file page.js

Segment (Phân đoạn route):
    Mỗi phần của URL path được ngăn cách bởi dấu gạch chéo (/) là 1 segment
    Ví dụ: trong path /dashboard/settings/profile
          - dashboard là 1 segment
          - settings là 1 segment
          - profile là 1 segment
    => Trong app route, mỗi thư mục bên trong folder app tương ứng với 1 segment trong URL
    Ví dụ:
        app/about => segment là about
        app/about/hehe => segment là hehe

- router.push('/path'): điều hướng đến 1 route mới và thêm 1 entry vào history stack của trình duyệt
  (người dùng có thể click nút back để về trang trước đó)

- router.replace('path'): điều hướng đến 1 route mới nhưng thay thế entry hiện tại trong history stack  
  (người dùng không thể back về trang vừa rời đi)

- router.back(): quay lại trang trước đó trong history

- router.forward(): đi tới trang tiếp theo trong history

- router.refresh(): làm mới route hiện tại. Nextjs sẽ refetch lại dữ liệu cho server component
  và re-render các Client component bị ảnh hưởng mà không làm mất state của các client component 
  không liên quan.

Server component: hoàn toàn được render phía server. Kết quả render sẽ được gửi xuống client 
      (HTML hoặc là 1 định dạng đặc biệt được gọi RSC payload)
                  Trong app router, thì mặc định tất cả component đều là server component.

    Đặc điểm: 
        + chạy trên server, không gửi javascript xuống frontend 
          (trừ khi nó render các client component con ở bên trong nó)
        + Không có state, không có lifecycle (vì chúng chỉ render 1 lần từ server cho mỗi request)
        + có thể async/await trực tiếp (ví dụ fetching data)
        + an toàn để chứa các logic nghiệp vụ nhạy cảm (ví dụ như truy vấn trực tiếp vào DB) vì code không
          được gửi xuống frontend
        + không sử dụng được các hook của react (useState, useEffect,...)
        + Không thể sử dụng các apis của trình duyệt (window, localstorage, ...)
        + khônng thể sử dụng các event handlers (onClick, onChange)

    Khi nào nên sử dụng: 
        + render dữ liệu trực tiếp từ backend ngay trên server để phục vụ SEO
        + truy cập trực tiếp vào tài nguyên backend nhạy cảm (database, filesystems) và không muốn để 
            cho frontend biết
        + Hiển thị nội dung tĩnh hoặc là nội dung ít thay đổi dựa trên tương tác người dùng
        + giữ các dependency lớn ở phía server mà không muốn gửi xuống client
        + tối ưu hoá bundle size và peformance

Client component: là React component được render dưới trình duyệt (sau quá trình hydration).
                Có thẻ tương các sự kiện (event handlers) và quản lý state phía client

        - hydration là gì?: là quá trình làm cho HTML tĩnh đã được render trước bên phía server 
                          trở nên có thể tương tác được:
            + server render và gửi HTML tĩnh về trình duyệt => user thấy giao diện ngay
            + Dưới trình duyệt (client), react bắt đầu quá trình hydrate bằng cách gán các event listeners
                vào các phần tử HTML (DOM elements)
            + => lúc này HTML tĩnh có thể tương tác được trong ứng dung react
      
      Đặc điểm:
          + Đánh dấu bằng "use client" ở phía đầu file (Để phân biệt CC và SC)
          + Được render trên phía server (pre-render) và sau đó hydrate dưới client
          + javascript của component này được gửi xuống client
          + có thể sử dụng state (useState), lifecycle effects (useEffect), và các hook khác của React
          + Có thể sử dụng các API của trình duyệt (localstorage, window, document.querySelector(),...)
          + Có thể sử dụng event handlers

      Khi nào nên sử dụng:
          + Có hỗ trợ tương tác người dùng (onClick, nhập form)
          + quản lý state bên phía client
          + dùng lifecycle
          + sử dụng apis của trình duyệt
          + sử dụng react hook, React context hoặc các thư viện quản lý state (redux)


Lợi ích của mô hình SC & CC:
  + hiệu suất: giảm lượng javascript gửi xuống client, cải thiệnt hời gian lúc load trang
              đến khi user tương tác được
  + await/async trực tiếp trong server component để fetching data
  + tối ưu hoá hơn: chỉ gửi code xuống client khi cần tương tác
